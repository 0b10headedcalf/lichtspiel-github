When It Breaks

It's a live performance — a USB cable will wiggle loose mid-show. The most interesting engineering in Lichtspiel is what happens when something fails and the show stays up anyway.

The metaphor: limp-home mode

When a sensor fails in a modern car, the engine doesn't just die on the motorway. It drops into limp-home mode: reduced power, fewer features, but it still drives you to a garage. Failing gracefully beats failing completely.

Lichtspiel is built entirely around this idea. The project documents call it "why the demo stays up." Every fancy layer can fall away, and a simpler one keeps the visuals running.

1
Ableton disconnects?

The visuals keep going — you can still drive everything from the keyboard and the controller.

2
The bridge dies?

The visual engine runs fully on its own in the browser — no bridge, no Ableton, no AI needed.

3
The bridge comes back?

The visuals notice and silently reconnect. No restart, no refresh.

💡
The vocabulary: "graceful degradation"

Graceful degradation is a hallmark of production-quality software. When you ask an AI to build something "robust for a live demo," asking specifically for graceful degradation — "if the server is down, the page should still work read-only" — is far more precise than "make it reliable."

Reconnecting, politely

When the bridge vanishes, the visuals don't hammer it with reconnect attempts. They wait a little, try, and if it fails, wait a bit longer next time — up to a ceiling. This is real code from the visual engine:

CODE 
private scheduleReconnect(): void {
  const delay = this.reconnectMs;
  this.reconnectMs = Math.min(this.reconnectMs * 1.6, 10000);
  window.setTimeout(() => {
    if (!this.closed) this.open();
  }, delay);
}
PLAIN ENGLISH

Schedule another attempt to reconnect.

Wait the current delay (it starts at 1 second).

Multiply the next wait by 1.6 — but never wait more than 10 seconds. Each failure backs off a little more.

After the wait, if we haven't been deliberately shut down, try to open the connection again.

⏳
The vocabulary: "exponential backoff"

Growing the wait time after each failure is called exponential backoff. Retrying instantly and forever would flood a struggling server and make things worse — backoff is the polite, standard fix. It's a great phrase to drop when asking an AI to handle flaky connections.

War story: the temperamental knob controller

Here's a real bug the team fought — and the kind of story that separates a working demo from a flaky one. One of the performer's controllers (an Arc 4) is a hand-built clone whose connection is fragile. The team learned three rules the hard way:

🤐

Talk less

Sending light updates to all rings 30 times a second flooded the clone's connection and disconnected it. Fix: only send a light change when it actually changed.

🧐

Trust nothing

The flaky device spat out a storm of "I'm here! / I'm gone!" notices. Reacting to each made it worse. Fix: ignore the storm and calmly poll "who's actually here?" on a timer.

🔌

Power-cycle when stuck

Sometimes the controller gets wedged — plugged in but invisible. The bridge detects this and restarts the controller service automatically. The software version of "turn it off and on again."

⚠️
The lesson for steering AI

An AI assistant will happily write "send all the LED updates every frame" — it looks correct and works on perfect hardware. The bug only appears with a flaky real device under load. You knowing to say "only send updates that changed — don't flood the device" is exactly the kind of intervention that rescues a demo. Real-world constraints are where human judgement still beats the autocomplete.

One more trick: reproducible randomness

The visuals are full of randomness — flickers, scattered shapes, organic drift. But "random" and "reproducible" sound like opposites. How do you debug a glitch you can't recreate?

Lichtspiel routes all its randomness through a seeded generator. Give it the same starting number (a "seed") and you get the exact same "random" sequence every time — so the exact same visual.

🎲
The vocabulary: a "seed" / "deterministic"

A seed makes randomness deterministic. This is gold for debugging: "it only glitches sometimes" becomes "seed 4827 reliably reproduces the glitch." If you ever fight a bug that won't reproduce, asking an AI "can we seed the randomness so this is repeatable?" is a pro move.
