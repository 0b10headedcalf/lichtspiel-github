Meet the Actors

Lichtspiel isn't one program — it's a small crew of five, each with exactly one job. Knowing who does what is how you tell an AI "fix it here, not there."

The metaphor: a film crew

A film set works because nobody does everyone's job. The camera operator doesn't write the script. The director doesn't splice the film. Each role is narrow, and that's the point — you can replace the camera operator without reshooting the script.

Lichtspiel is built the same way. Five separate programs, each a specialist. Here's the crew:

🎚️
The Max device — embedded inside Ableton

A small panel that lives inside the music software and watches the song: tempo, which clip is playing, when a section starts. Built with Max for Live.

🔀
The bridge — the dispatcher

A Node.js program that sits in the middle and relays messages between everyone. The traffic hub. (Whole next module is about how it routes.)

🌀
The p5 runtime — the screen

The visual engine that actually draws the tunnel, running in a browser using p5.js.

🧠
The ML service — the casting director

A Python helper that, given what the music is doing, suggests which visual would fit best.

🎛️
The monome — the control rig

The physical grid-and-knob hardware in the performer's hands. We met it in Module 1; Module 5 is all about it.

Where each actor lives in the project

Open the project folder and the crew is right there — one folder per specialist. This layout is the architecture:

apps/ The three running programs
live-bridge/The dispatcher (Node)
p5-runtime/The visual engine (browser) — "the heart of the demo"
ml-service/The casting director (Python)
max/The device that lives inside Ableton
packages/schemas/The shared script everyone reads from (Module 3)
💡
The vocabulary: "separation of concerns"

Splitting a system into focused parts, each responsible for one thing, is called separation of concerns — and a multi-program setup like this is often called a microservices architecture. When you tell an AI "the visual is wrong, look in the p5 runtime, not the bridge," this is the knowledge that makes the instruction precise.

A rule the whole crew obeys

One design decision shapes everything. The team wrote it down and locked it:

📌
"Max is the Live-native shell, not the brain."

The part inside Ableton stays deliberately dumb — it just reports what the song is doing. All the heavy thinking lives in the bridge, Python, and the visuals. Why? So the performance never depends on the most fragile link, and the visual engine can even run completely on its own.

Keep that in mind — it's the reason the show stays up even when pieces fail (Module 6).

How the crew actually wires up

When the bridge starts up, it literally plugs the actors into itself. Here's the moment a physical controller gets connected — real code from the bridge's startup file:

CODE 
onDeviceAttached: (d) => {
  server.ingest(wire('device.attached', d));
  server.setMonomeConnected(true);
},
PLAIN ENGLISH

"When a monome controller gets plugged in, do this:"

Wrap the news as a tidy message ("device.attached") and feed it into the dispatcher, exactly like any other message.

Flip an internal switch so the rest of the system knows hardware is now present.

Done. Notice the dispatcher doesn't care who sent the news — every actor's news enters through the same door.
