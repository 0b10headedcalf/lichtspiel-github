The Monome Instrument

The hardware the performer's hands live on — a grid of buttons and rings of knobs. The clever part: the same code plays a big controller or a small one, automatically.

The metaphor: a mixing desk for visuals

Picture the row of vertical sliders on a sound mixing desk — push a slider up, that channel gets louder. The monome grid is exactly this, but each column is a slider for a visual dial instead of a track's volume.

Press a cell high up in a column → that dial goes high. Press low → it goes low. The lit cells below your finger fill in like a level meter. Eight columns, eight dials, under your fingertips.

col 0motion — how fast things move
col 1density — how busy the image is
col 6palette — the colour scheme
col 7strobe — the flicker / damage

The rings ("arc") handle the smooth, continuous controls — twist for semantic distance, mutation, and more.

One press, in real code

Here's the exact logic that runs when you press a grid cell. Notice it reads the controller's size every single press — it never assumes:

CODE 
onGrid(e: GridKeyEvent): void {
  if (e.state !== 1) return; // act on press
  const grid = getSetup().grid;
  const rows = grid?.rows ?? 8;
  if (e.x >= COLUMN_AXES.length) return;
  const axis = COLUMN_AXES[e.x];
  if (!axis) return;
  const value = rows > 1 ? (rows - 1 - e.y) / (rows - 1) : 1;
  h.setParam(axis, clamp01(value));
},
PLAIN ENGLISH

When a grid cell changes, run this — but only on the press, ignore the release.

Ask the currently connected controller how many rows it has (default to 8 if unsure).

If you pressed a column beyond the eight dials, do nothing — extra columns aren't faders.

Figure out which dial this column controls.

Turn the row you pressed into a 0-to-1 value — top row = 1, bottom = 0 — scaled to this controller's height.

Set that dial (clamped safely to 0–1, of course).

💡
"Never hard-code the size"

That line rows ?? 8 and dividing by rows - 1 is the whole trick. The code asks the hardware its dimensions instead of assuming. This is why the same press logic works on a short controller and a tall one — a principle the project states bluntly: "Never hard-code grid size or encoder count; read it from the connected device."

The performer owns two sizes of each

There isn't one monome — there are four, in two sizes. The app must play all of them:

⬜

Grid 64

8×8 buttons. Lights are on/off only ("monobright").

▦

Grid 128

8×16 buttons — twice as wide. Lights have 16 brightness levels.

◐

Arc 2

2 knob rings.

◕

Arc 4

4 knob rings — double the continuous controls.

The key idea: capability folding

This is the cleverest concept in the whole project. Think of a symphony arranged for a smaller piano. The melody survives — it's folded onto the keys you actually have. Lichtspiel arranges its controls to fit whatever controller is plugged in.

Why this is rare and valuable

Most software is written for one exact device. Lichtspiel writes the logic once against "whatever capabilities this controller reports," so a single codebase fits four controllers. The project even has a rule against the lazy shortcut: "Don't port 1:1 — compress" a big layout down to a small one, rather than just ignoring the difference.

The lights talk back: the digital twin

An instrument you can't see is hard to play. So the controller's buttons and rings light up to mirror the current dials — and there's an on-screen "digital twin" that shows the exact same lights.

The slick part: the screen twin and the real hardware are driven by the same brightness code. So "what the twin shows" always equals "what the LEDs show." If you have no hardware at all, you can click the twin and it behaves identically — perfect for practising or for a demo.

🪞
One source of truth, two displays

Having one piece of logic feed both a real device and its on-screen mirror is a powerful pattern — it makes the two impossible to disagree. When you want an AI to build a "preview that always matches the real thing," this is the shape to ask for: a single source feeding both views.
