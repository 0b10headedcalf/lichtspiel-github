The Shared Language

Five programs, written in three languages, on two devices — yet they collaborate perfectly. The secret is one tiny, strict, shared vocabulary.

The metaphor: shipping containers

Before the 1950s, loading a ship meant manually wrestling barrels, crates, and sacks of every shape. Then someone standardized the box: one steel container size that every crane, truck, train, and ship could handle — without ever caring what's inside.

Lichtspiel's actors agreed on the same trick. They don't pass around freeform data. They pass standard containers. Any actor can handle one without knowing who packed it.

📦
The vocabulary: a "schema" / "contract"

A schema is the agreed-upon shape of a container. All of Lichtspiel's schemas live in one shared folder — packages/schemas — described in the code as "the single source of truth." Define it once; everyone imports it; nobody redefines it.

Container #1: the dashboard of dials

The most important container is the VisualParamVector — think of it as a standard dashboard with sixteen labelled dials. Every visual understands these same dials. Here are some of them:

🌪️

density · motion · turbulence

How busy, how fast, how chaotic the image is.

🎨

palette · contrast · lineWeight

Colour scheme and the "look" of the strokes.

📷

cameraDepth · rotationX/Y/Z

Where the virtual camera is and how it's turned.

🧭

semanticDistance · mutationAmount

How far to roam from the current look, and how wild the changes get.

Here's the actual container definition, with its sensible "middle" defaults — this is exactly what's in the code:

CODE 
export const DEFAULT_PARAMS: VisualParamVector = Object.freeze({
  sceneId: 'minimalPulse',
  density: 0.5,
  motion: 0.5,
  turbulence: 0.5,
  symmetry: 0.5,
  strobe: 0.0,
  // ...ten more dials, all 0..1...
});
PLAIN ENGLISH

Define the standard dashboard and "freeze" it so nobody can secretly change the template.

sceneId is the one text field — it names which visual this dashboard is for.

Every other field is a number starting at a neutral middle (0.5) or off (0.0)...

...so a fresh visual always boots up looking reasonable, before you touch a single knob.

The one rule that prevents chaos: 0 to 1

Every dial is a number between 0 and 1. Not 0–100, not 0–255, not "low/medium/high." Just 0 to 1. This single convention means a knob, a song's tempo, and an AI suggestion can all drive the same dial without unit confusion.

And the code enforces it. Any value that wanders out of bounds gets snapped back:

CODE 
export function clamp01(x: number): number {
  if (Number.isNaN(x)) return 0;
  if (x < 0) return 0;
  if (x > 1) return 1;
  return x;
}
PLAIN ENGLISH

Take any number and force it into the 0-to-1 range.

If it's broken ("not a number"), treat it as 0 — never let garbage through.

Below 0? Snap up to 0. Above 1? Snap down to 1.

Otherwise it's already valid — pass it through unchanged.

🛡️
This is "defensive programming"

Tiny guard functions like clamp01 are how robust software stays robust. Instead of hoping every caller behaves, the value is sanitized at the door. When you ask an AI to "make this bulletproof," asking it to "clamp and validate inputs" is exactly the kind of precise instruction that pays off.

Container #2: the shipping label

Containers also need a label so the dispatcher knows what they are and what to do with them. Every message between actors carries the same four-part label, called the wire protocol:

vVersion number — so old and new actors don't misread each other.
tsTimestamp — exactly when it was sent.
typeWhat kind of message: "params.update", "scene.select", "monome.event"…
payloadThe actual cargo — e.g. the dashboard of dials.

In the code, every possible message is one labelled line. Here's the one for a dial change:

CODE 
| { v: 1; ts: number; type: 'params.update'; payload: Partial<VisualParamVector> }
PLAIN ENGLISH

"A valid message can be: version 1, a timestamp, type params.update, carrying some of the dials." (Partial = you don't have to send all sixteen — just the ones that changed.)
