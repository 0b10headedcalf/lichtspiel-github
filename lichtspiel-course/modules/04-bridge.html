The Bridge

The dispatcher in the middle. It receives every message from everyone, checks it, and decides exactly who needs to hear it. Most "it's not showing up" bugs live here.

The metaphor: a mail sorting office

Mail pours in from every neighbourhood. A sorting office does three things, endlessly: check each envelope has a valid address, sort it into the right destination bin, and toss anything undeliverable into the dead-letter pile (with a note in the log).

The bridge — one Node program in apps/live-bridge — is that office. The Max device, the controllers, the command line, and the visuals all drop mail here. The bridge never creates the content; it just moves it to the right place, safely.

✅
Check

Does this message match a known, valid shape? If not, drop it.

📬
Sort

Send it only to the actors that care — the screens, the controllers, or both.

🗒️
Log

Write down what happened, especially the rejects, so you can debug later.

"Sort by who you are"

When any actor connects, the very first thing it does is announce its role: "I'm a screen" (p5), "I'm the Ableton device" (max), or "I'm a command-line tool" (cli). The bridge remembers this, and uses it to address every later message.

This is why a song-state update goes only to the screens, and a button-press goes to both the screens and the Ableton device — the bridge sorts by who's listening.

💡
The vocabulary: a "WebSocket" and "broadcast"

The actors stay connected over a WebSocket — a live, always-open pipe. Sending one message to many recipients at once is called a broadcast. When a feature 'works for one screen but not the second monitor,' the broadcast logic is the first place to look.

Check, then sort — in real code

That whole "check, then sort, else log and drop" pattern is just a few lines. Here's how the bridge handles a button-or-knob event from a controller:

CODE 
if (isType(m, 'monome.event')) {
  const v = validate('MonomeEvent', m.payload);
  if (v.valid) this.broadcast(['p5', 'max'], m);
  else logger.warn('monome.event', { source: src, valid: false, error: v.error });
  return;
}
PLAIN ENGLISH

"If this message is a controller event..."

...check it matches the known, valid shape for one.

If valid: broadcast it to the screens AND the Ableton device (both want to know about button presses).

If not: write a warning to the log — including why it failed — and quietly drop it.

Either way, stop here. (Notice: a bad message can never reach the visuals.)

A clever touch: catching up late arrivals

Here's a subtle problem the bridge solves. A controller getting plugged in is a one-time announcement. So what about a screen that connects after that — does it never learn there's hardware attached?

The bridge keeps a small memory of "standing notices" — the currently attached devices and the latest song snapshot — and replays them to any screen that joins late. Like a new resident getting handed the notices already posted on the board.

🔁
Why this matters for you

"It works when I start everything in the right order, but not if I refresh the page" is one of the most common real-world bugs. The fix is almost always this exact pattern: replay current state to late joiners instead of assuming everyone was present for the original announcement. Now you can name it when you ask an AI to fix it.
