Inside Lichtspiel
01

What Lichtspiel Does

A musician plays Ableton. The visuals listen, and answer. Let's follow one twist of a knob all the way through the machine.

The thirty-second pitch

Lichtspiel turns the music you're already making in Ableton Live into live, code-drawn visuals — and lets you play those visuals with your hands, on physical knob-and-button controllers.

It is not a video player. There are no video files. Every frame is drawn live by code that reacts, in real time, to the structure of your song.

🎬
Why the name?

Lichtspiel is German for "play of light" — an old word for cinema. The signature visual is a morphing tunnel inspired by Walter Ruttmann's 1920s abstract films. Fitting, for a tool that turns sound into moving light.

The metaphor: a marionette

Picture a marionette. You hold a control bar. You tilt it — and across the stage, the puppet's arm rises. You never touch the puppet directly; you move a control, and the motion travels down the strings.

Lichtspiel works exactly like this. Your hands move a physical controller (the bar). That motion travels through a chain of software (the strings). At the far end, the visual (the puppet) responds. The whole course is about what those strings are made of.

🎛️
The control bar — a "monome"

A grid of buttons and a ring of knobs. This is what the performer's hands actually touch.

🧵
The strings — software passing messages

Several programs hand a small message down the line, each doing one job.

🌀
The puppet — the visual on screen

A tunnel of light that warps, twists, and breathes in response.

The actual code for that twist

That middle step — "look up what the knob controls, nudge that dial" — is astonishingly small. This is the real code that runs every time you turn a ring knob:

CODE 
onArcDelta(e: ArcDeltaEvent): void {
  const step = e.delta / 64; // arc rings are 64 LEDs (v3 convention)
  const axis = ARC_AXES[e.encoder];
  if (axis) h.nudgeParam(axis, step);
},
PLAIN ENGLISH

When a knob ("arc") moves, run this.

The knob reports a raw amount; divide by 64 (a ring has 64 lights) to turn it into a gentle 0-to-1-sized nudge.

Look up which dial this knob is wired to. Knob 0 → "semantic distance", knob 1 → "mutation amount", and so on.

If that knob is wired to a dial, nudge that dial by the small amount. That's the whole move.

💡
The big idea hiding in five lines

The knob doesn't know about tunnels. It just nudges a numbered dial. The visual doesn't know about knobs — it just reads dials. Neither side knows the other exists. That separation is the entire architecture, and it's why this app is so flexible. We'll unpack it across the next five modules.

02

Meet the Actors

Lichtspiel isn't one program — it's a small crew of five, each with exactly one job. Knowing who does what is how you tell an AI "fix it here, not there."

The metaphor: a film crew

A film set works because nobody does everyone's job. The camera operator doesn't write the script. The director doesn't splice the film. Each role is narrow, and that's the point — you can replace the camera operator without reshooting the script.

Lichtspiel is built the same way. Five separate programs, each a specialist. Here's the crew:

🎚️
The Max device — embedded inside Ableton

A small panel that lives inside the music software and watches the song: tempo, which clip is playing, when a section starts. Built with Max for Live.

🔀
The bridge — the dispatcher

A Node.js program that sits in the middle and relays messages between everyone. The traffic hub. (Whole next module is about how it routes.)

🌀
The p5 runtime — the screen

The visual engine that actually draws the tunnel, running in a browser using p5.js.

🧠
The ML service — the casting director

A Python helper that, given what the music is doing, suggests which visual would fit best.

🎛️
The monome — the control rig

The physical grid-and-knob hardware in the performer's hands. We met it in Module 1; Module 5 is all about it.

Where each actor lives in the project

Open the project folder and the crew is right there — one folder per specialist. This layout is the architecture:

apps/ The three running programs
live-bridge/The dispatcher (Node)
p5-runtime/The visual engine (browser) — "the heart of the demo"
ml-service/The casting director (Python)
max/The device that lives inside Ableton
packages/schemas/The shared script everyone reads from (Module 3)
💡
The vocabulary: "separation of concerns"

Splitting a system into focused parts, each responsible for one thing, is called separation of concerns — and a multi-program setup like this is often called a microservices architecture. When you tell an AI "the visual is wrong, look in the p5 runtime, not the bridge," this is the knowledge that makes the instruction precise.

A rule the whole crew obeys

One design decision shapes everything. The team wrote it down and locked it:

📌
"Max is the Live-native shell, not the brain."

The part inside Ableton stays deliberately dumb — it just reports what the song is doing. All the heavy thinking lives in the bridge, Python, and the visuals. Why? So the performance never depends on the most fragile link, and the visual engine can even run completely on its own.

Keep that in mind — it's the reason the show stays up even when pieces fail (Module 6).

How the crew actually wires up

When the bridge starts up, it literally plugs the actors into itself. Here's the moment a physical controller gets connected — real code from the bridge's startup file:

CODE 
onDeviceAttached: (d) => {
  server.ingest(wire('device.attached', d));
  server.setMonomeConnected(true);
},
PLAIN ENGLISH

"When a monome controller gets plugged in, do this:"

Wrap the news as a tidy message ("device.attached") and feed it into the dispatcher, exactly like any other message.

Flip an internal switch so the rest of the system knows hardware is now present.

Done. Notice the dispatcher doesn't care who sent the news — every actor's news enters through the same door.

03

The Shared Language

Five programs, written in three languages, on two devices — yet they collaborate perfectly. The secret is one tiny, strict, shared vocabulary.

The metaphor: shipping containers

Before the 1950s, loading a ship meant manually wrestling barrels, crates, and sacks of every shape. Then someone standardized the box: one steel container size that every crane, truck, train, and ship could handle — without ever caring what's inside.

Lichtspiel's actors agreed on the same trick. They don't pass around freeform data. They pass standard containers. Any actor can handle one without knowing who packed it.

📦
The vocabulary: a "schema" / "contract"

A schema is the agreed-upon shape of a container. All of Lichtspiel's schemas live in one shared folder — packages/schemas — described in the code as "the single source of truth." Define it once; everyone imports it; nobody redefines it.

Container #1: the dashboard of dials

The most important container is the VisualParamVector — think of it as a standard dashboard with sixteen labelled dials. Every visual understands these same dials. Here are some of them:

🌪️

density · motion · turbulence

How busy, how fast, how chaotic the image is.

🎨

palette · contrast · lineWeight

Colour scheme and the "look" of the strokes.

📷

cameraDepth · rotationX/Y/Z

Where the virtual camera is and how it's turned.

🧭

semanticDistance · mutationAmount

How far to roam from the current look, and how wild the changes get.

Here's the actual container definition, with its sensible "middle" defaults — this is exactly what's in the code:

CODE 
export const DEFAULT_PARAMS: VisualParamVector = Object.freeze({
  sceneId: 'minimalPulse',
  density: 0.5,
  motion: 0.5,
  turbulence: 0.5,
  symmetry: 0.5,
  strobe: 0.0,
  // ...ten more dials, all 0..1...
});
PLAIN ENGLISH

Define the standard dashboard and "freeze" it so nobody can secretly change the template.

sceneId is the one text field — it names which visual this dashboard is for.

Every other field is a number starting at a neutral middle (0.5) or off (0.0)...

...so a fresh visual always boots up looking reasonable, before you touch a single knob.

The one rule that prevents chaos: 0 to 1

Every dial is a number between 0 and 1. Not 0–100, not 0–255, not "low/medium/high." Just 0 to 1. This single convention means a knob, a song's tempo, and an AI suggestion can all drive the same dial without unit confusion.

And the code enforces it. Any value that wanders out of bounds gets snapped back:

CODE 
export function clamp01(x: number): number {
  if (Number.isNaN(x)) return 0;
  if (x < 0) return 0;
  if (x > 1) return 1;
  return x;
}
PLAIN ENGLISH

Take any number and force it into the 0-to-1 range.

If it's broken ("not a number"), treat it as 0 — never let garbage through.

Below 0? Snap up to 0. Above 1? Snap down to 1.

Otherwise it's already valid — pass it through unchanged.

🛡️
This is "defensive programming"

Tiny guard functions like clamp01 are how robust software stays robust. Instead of hoping every caller behaves, the value is sanitized at the door. When you ask an AI to "make this bulletproof," asking it to "clamp and validate inputs" is exactly the kind of precise instruction that pays off.

Container #2: the shipping label

Containers also need a label so the dispatcher knows what they are and what to do with them. Every message between actors carries the same four-part label, called the wire protocol:

vVersion number — so old and new actors don't misread each other.
tsTimestamp — exactly when it was sent.
typeWhat kind of message: "params.update", "scene.select", "monome.event"…
payloadThe actual cargo — e.g. the dashboard of dials.

In the code, every possible message is one labelled line. Here's the one for a dial change:

CODE 
| { v: 1; ts: number; type: 'params.update'; payload: Partial<VisualParamVector> }
PLAIN ENGLISH

"A valid message can be: version 1, a timestamp, type params.update, carrying some of the dials." (Partial = you don't have to send all sixteen — just the ones that changed.)

04

The Bridge

The dispatcher in the middle. It receives every message from everyone, checks it, and decides exactly who needs to hear it. Most "it's not showing up" bugs live here.

The metaphor: a mail sorting office

Mail pours in from every neighbourhood. A sorting office does three things, endlessly: check each envelope has a valid address, sort it into the right destination bin, and toss anything undeliverable into the dead-letter pile (with a note in the log).

The bridge — one Node program in apps/live-bridge — is that office. The Max device, the controllers, the command line, and the visuals all drop mail here. The bridge never creates the content; it just moves it to the right place, safely.

✅
Check

Does this message match a known, valid shape? If not, drop it.

📬
Sort

Send it only to the actors that care — the screens, the controllers, or both.

🗒️
Log

Write down what happened, especially the rejects, so you can debug later.

"Sort by who you are"

When any actor connects, the very first thing it does is announce its role: "I'm a screen" (p5), "I'm the Ableton device" (max), or "I'm a command-line tool" (cli). The bridge remembers this, and uses it to address every later message.

This is why a song-state update goes only to the screens, and a button-press goes to both the screens and the Ableton device — the bridge sorts by who's listening.

💡
The vocabulary: a "WebSocket" and "broadcast"

The actors stay connected over a WebSocket — a live, always-open pipe. Sending one message to many recipients at once is called a broadcast. When a feature 'works for one screen but not the second monitor,' the broadcast logic is the first place to look.

Check, then sort — in real code

That whole "check, then sort, else log and drop" pattern is just a few lines. Here's how the bridge handles a button-or-knob event from a controller:

CODE 
if (isType(m, 'monome.event')) {
  const v = validate('MonomeEvent', m.payload);
  if (v.valid) this.broadcast(['p5', 'max'], m);
  else logger.warn('monome.event', { source: src, valid: false, error: v.error });
  return;
}
PLAIN ENGLISH

"If this message is a controller event..."

...check it matches the known, valid shape for one.

If valid: broadcast it to the screens AND the Ableton device (both want to know about button presses).

If not: write a warning to the log — including why it failed — and quietly drop it.

Either way, stop here. (Notice: a bad message can never reach the visuals.)

A clever touch: catching up late arrivals

Here's a subtle problem the bridge solves. A controller getting plugged in is a one-time announcement. So what about a screen that connects after that — does it never learn there's hardware attached?

The bridge keeps a small memory of "standing notices" — the currently attached devices and the latest song snapshot — and replays them to any screen that joins late. Like a new resident getting handed the notices already posted on the board.

🔁
Why this matters for you

"It works when I start everything in the right order, but not if I refresh the page" is one of the most common real-world bugs. The fix is almost always this exact pattern: replay current state to late joiners instead of assuming everyone was present for the original announcement. Now you can name it when you ask an AI to fix it.

05

The Monome Instrument

The hardware the performer's hands live on — a grid of buttons and rings of knobs. The clever part: the same code plays a big controller or a small one, automatically.

The metaphor: a mixing desk for visuals

Picture the row of vertical sliders on a sound mixing desk — push a slider up, that channel gets louder. The monome grid is exactly this, but each column is a slider for a visual dial instead of a track's volume.

Press a cell high up in a column → that dial goes high. Press low → it goes low. The lit cells below your finger fill in like a level meter. Eight columns, eight dials, under your fingertips.

col 0motion — how fast things move
col 1density — how busy the image is
col 6palette — the colour scheme
col 7strobe — the flicker / damage

The rings ("arc") handle the smooth, continuous controls — twist for semantic distance, mutation, and more.

One press, in real code

Here's the exact logic that runs when you press a grid cell. Notice it reads the controller's size every single press — it never assumes:

CODE 
onGrid(e: GridKeyEvent): void {
  if (e.state !== 1) return; // act on press
  const grid = getSetup().grid;
  const rows = grid?.rows ?? 8;
  if (e.x >= COLUMN_AXES.length) return;
  const axis = COLUMN_AXES[e.x];
  if (!axis) return;
  const value = rows > 1 ? (rows - 1 - e.y) / (rows - 1) : 1;
  h.setParam(axis, clamp01(value));
},
PLAIN ENGLISH

When a grid cell changes, run this — but only on the press, ignore the release.

Ask the currently connected controller how many rows it has (default to 8 if unsure).

If you pressed a column beyond the eight dials, do nothing — extra columns aren't faders.

Figure out which dial this column controls.

Turn the row you pressed into a 0-to-1 value — top row = 1, bottom = 0 — scaled to this controller's height.

Set that dial (clamped safely to 0–1, of course).

💡
"Never hard-code the size"

That line rows ?? 8 and dividing by rows - 1 is the whole trick. The code asks the hardware its dimensions instead of assuming. This is why the same press logic works on a short controller and a tall one — a principle the project states bluntly: "Never hard-code grid size or encoder count; read it from the connected device."

The performer owns two sizes of each

There isn't one monome — there are four, in two sizes. The app must play all of them:

⬜

Grid 64

8×8 buttons. Lights are on/off only ("monobright").

▦

Grid 128

8×16 buttons — twice as wide. Lights have 16 brightness levels.

◐

Arc 2

2 knob rings.

◕

Arc 4

4 knob rings — double the continuous controls.

The key idea: capability folding

This is the cleverest concept in the whole project. Think of a symphony arranged for a smaller piano. The melody survives — it's folded onto the keys you actually have. Lichtspiel arranges its controls to fit whatever controller is plugged in.

→
2

→
3

🎹
Why this is rare and valuable

Most software is written for one exact device. Lichtspiel writes the logic once against "whatever capabilities this controller reports," so a single codebase fits four controllers. The project even has a rule against the lazy shortcut: "Don't port 1:1 — compress" a big layout down to a small one, rather than just ignoring the difference.

The lights talk back: the digital twin

An instrument you can't see is hard to play. So the controller's buttons and rings light up to mirror the current dials — and there's an on-screen "digital twin" that shows the exact same lights.

The slick part: the screen twin and the real hardware are driven by the same brightness code. So "what the twin shows" always equals "what the LEDs show." If you have no hardware at all, you can click the twin and it behaves identically — perfect for practising or for a demo.

🪞
One source of truth, two displays

Having one piece of logic feed both a real device and its on-screen mirror is a powerful pattern — it makes the two impossible to disagree. When you want an AI to build a "preview that always matches the real thing," this is the shape to ask for: a single source feeding both views.

06

When It Breaks

It's a live performance — a USB cable will wiggle loose mid-show. The most interesting engineering in Lichtspiel is what happens when something fails and the show stays up anyway.

The metaphor: limp-home mode

When a sensor fails in a modern car, the engine doesn't just die on the motorway. It drops into limp-home mode: reduced power, fewer features, but it still drives you to a garage. Failing gracefully beats failing completely.

Lichtspiel is built entirely around this idea. The project documents call it "why the demo stays up." Every fancy layer can fall away, and a simpler one keeps the visuals running.

1
Ableton disconnects?

The visuals keep going — you can still drive everything from the keyboard and the controller.

2
The bridge dies?

The visual engine runs fully on its own in the browser — no bridge, no Ableton, no AI needed.

3
The bridge comes back?

The visuals notice and silently reconnect. No restart, no refresh.

💡
The vocabulary: "graceful degradation"

Graceful degradation is a hallmark of production-quality software. When you ask an AI to build something "robust for a live demo," asking specifically for graceful degradation — "if the server is down, the page should still work read-only" — is far more precise than "make it reliable."

Reconnecting, politely

When the bridge vanishes, the visuals don't hammer it with reconnect attempts. They wait a little, try, and if it fails, wait a bit longer next time — up to a ceiling. This is real code from the visual engine:

CODE 
private scheduleReconnect(): void {
  const delay = this.reconnectMs;
  this.reconnectMs = Math.min(this.reconnectMs * 1.6, 10000);
  window.setTimeout(() => {
    if (!this.closed) this.open();
  }, delay);
}
PLAIN ENGLISH

Schedule another attempt to reconnect.

Wait the current delay (it starts at 1 second).

Multiply the next wait by 1.6 — but never wait more than 10 seconds. Each failure backs off a little more.

After the wait, if we haven't been deliberately shut down, try to open the connection again.

⏳
The vocabulary: "exponential backoff"

Growing the wait time after each failure is called exponential backoff. Retrying instantly and forever would flood a struggling server and make things worse — backoff is the polite, standard fix. It's a great phrase to drop when asking an AI to handle flaky connections.

War story: the temperamental knob controller

Here's a real bug the team fought — and the kind of story that separates a working demo from a flaky one. One of the performer's controllers (an Arc 4) is a hand-built clone whose connection is fragile. The team learned three rules the hard way:

🤐

Talk less

Sending light updates to all rings 30 times a second flooded the clone's connection and disconnected it. Fix: only send a light change when it actually changed.

🧐

Trust nothing

The flaky device spat out a storm of "I'm here! / I'm gone!" notices. Reacting to each made it worse. Fix: ignore the storm and calmly poll "who's actually here?" on a timer.

🔌

Power-cycle when stuck

Sometimes the controller gets wedged — plugged in but invisible. The bridge detects this and restarts the controller service automatically. The software version of "turn it off and on again."

⚠️
The lesson for steering AI

An AI assistant will happily write "send all the LED updates every frame" — it looks correct and works on perfect hardware. The bug only appears with a flaky real device under load. You knowing to say "only send updates that changed — don't flood the device" is exactly the kind of intervention that rescues a demo. Real-world constraints are where human judgement still beats the autocomplete.

One more trick: reproducible randomness

The visuals are full of randomness — flickers, scattered shapes, organic drift. But "random" and "reproducible" sound like opposites. How do you debug a glitch you can't recreate?

Lichtspiel routes all its randomness through a seeded generator. Give it the same starting number (a "seed") and you get the exact same "random" sequence every time — so the exact same visual.

🎲
The vocabulary: a "seed" / "deterministic"

A seed makes randomness deterministic. This is gold for debugging: "it only glitches sometimes" becomes "seed 4827 reliably reproduces the glitch." If you ever fight a bug that won't reproduce, asking an AI "can we seed the randomness so this is repeatable?" is a pro move.
