# Max for Live — Phase 3 setup, test & build recipe

The pieces that exist now:

- **`js/live_api_helpers.js`** — reads the Live Object Model via `LiveAPI` on
  every `bang` and emits a normalized `LiveSessionState` as one JSON symbol.
  Guarded so it never errors (outside Live it emits the default state).
- **`patches/lichtspiel_probe.maxpat`** — a generated, openable Max patch:
  `loadbang`/`metro 250` → `js live_api_helpers.js` → `prepend /lichtspiel/state`
  → `udpsend 127.0.0.1 7400`. (Generated by `build_patches.py` via MaxPyLang —
  see below. The colocated `patches/live_api_helpers.js` is a build copy so the
  patch finds it without a search-path edit.)
- **bridge OSC receiver** (`apps/live-bridge/src/oscRouter.ts`) — decodes
  `/lichtspiel/state|scene|param` on UDP 7400 → forwards to the p5 runtime.
  Verified end-to-end by `pnpm --filter @lichtspiel/live-bridge test:osc`.

The OSC contract this uses is the Live side of `docs/monome.md`'s serialosc
section: prefix `/lichtspiel`, app→bridge UDP `7400`.

## Test A — OSC plumbing, no Ableton (do this first)

Proves Max → bridge → p5 without needing the Live API.

1. `pnpm dev:bridge` then `pnpm dev:p5` (browser shows the runtime).
2. Open **`patches/lichtspiel_probe.maxpat`** in standalone **Max** and turn the
   patch on (the `loadbang`/`metro` start automatically; click the `1` message
   if needed). It sends a default `LiveSessionState` every 250 ms.
3. The p5 pill should flip to **"bridge connected"**; the bridge log prints
   `live.state … @ 120bpm`. (Standalone Max has no `LiveAPI`, so clip fields are
   empty — that's expected; this test is only the transport.)

## Test B — real Live data as a Max for Live device

`.amxd` packaging needs the Max GUI (human-in-the-loop), so assemble the device
once:

1. In Live, drop a **Max Audio Effect** on a track → click the edit (pencil) icon.
2. Recreate the probe objects (or paste from `lichtspiel_probe.maxpat`):
   `js live_api_helpers.js` → `prepend /lichtspiel/state` → `udpsend 127.0.0.1 7400`,
   plus `live.thisdevice` → `js` (fires when the Live set is ready) and a
   `metro 250` (toggled on) → `js` for a heartbeat.
3. Make sure Max can find the script: keep `live_api_helpers.js` in the device
   folder, **or** add `max/js/` to **Options → File Preferences** search path.
4. Save the device as `devices/LichtspielHub.amxd`.
5. Run `pnpm dev:bridge` + `pnpm dev:p5`. Select different clips/scenes in Live →
   the bridge log shows the clip name + tempo, and p5 receives `live.state`.
   (Phase 5 wires that state through retrieval to actually pick a scene; for now
   you can also drive scenes with `pnpm send scene …`.)

If anything in the patch misbehaves, paste the Max console error back and I'll
adjust the generator.

## Test C — device controls → p5 params (3a)

Makes the loop bidirectional: knobs in the device push visual params to p5.
`patches/lichtspiel_controls.maxpat` has 6 `live.dial`s → `[prepend
/lichtspiel/param <name>]` → `[udpsend 127.0.0.1 7400]` (density, motion,
palette, cameraDepth, mutationAmount, semanticDistance) + 5 scene message
buttons → `[prepend /lichtspiel/scene]`. The bridge already maps `/param` →
params.update and `/scene` → scene.select (both verified live).

1. In the **device editor** (same one from Test B): **File ▸ Open** →
   `…/max/patches/lichtspiel_controls.maxpat`.
2. **Cmd+A, Cmd+C** there → click the device canvas → **Cmd+V** (it pastes
   *alongside* the probe; both share the `udpsend`).
3. **Set the dial range:** drag-select all 6 `live.dial`s → open the Inspector
   (Cmd+I) → set **Range/Enum** to `0.` … `1.` (one edit applies to all).
   Otherwise dials send 0–127 and saturate the param at 1.
4. **Cmd+S**. Now turning a dial in the device moves that param in the p5 HUD;
   clicking a scene button switches the visual.

(The `live.dial`s auto-register as mappable M4L parameters — the start of the
real device UI; 3b adds layout/labels.)

## Regenerating the patches

```bash
python3 -m venv max/.venv
max/.venv/bin/pip install maxpylang
max/.venv/bin/python max/build_patches.py    # writes lichtspiel_probe.maxpat + lichtspiel_controls.maxpat (+ js copy)
```

`build_patches.py` uses **MaxPyLang** (MIT) to emit valid patcher JSON so we
don't hand-author it. MaxPyLang doesn't carry the `live.*` object metadata, so
`live.thisdevice` + the device UI are added in the GUI (above). The
[MAX-MSP_CC_Framework](https://github.com/taylorbrook/MAX-MSP_CC_Framework) is a
good Claude-assisted patch-authoring aid if we iterate further.

## Building out `LichtspielHub.amxd` (full device, later)

- Expose Live params (`live.dial`/`live.text`): Enable, Auto Follow Clip, Scene
  Lock, Visual Scene, Mutation Amount, Semantic Distance, Density, Motion, Color,
  Camera, Reset, Surprise. Route them out as `/lichtspiel/param <name> <float>`
  (the bridge already maps `/param` → `params.update`) or `/lichtspiel/scene <id>`.
- Compact status UI (Status / Source / Visual / Macros / Diagnostics — spec §15).
- Keep the patch **thin**: observe + format + forward. No retrieval/rendering in Max.

## Live API observation priority (spec §7.2)

transport (tempo, playing, beat/bar) → selection (track/scene/clip name, color,
type) → playing clip per track → clip content (MIDI summary, audio path, loop) →
devices/macros → control surfaces (minimal). `live_api_helpers.js` currently
covers transport + selected track/scene/clip; extend `readState()` for the rest.

## Acceptance (Phase 3)

Changing the selected clip / toggling transport updates the bridge log; device
loads with no missing deps; M4L manual controls move p5 params; device works
with the ML service offline.
